## Previous Approach — Pivoted (JavaFX Simulator)

The initial implementation was a JavaFX desktop dashboard (`BruteForceDetector.java`) that simulated brute force detection. It used a sliding window algorithm backed by `HashMap<String, LinkedList<Long>>` and a `HashSet<String>` for blocked IPs.

**Why it was abandoned:** The core weakness was that the "detection" processed fake data generated by the application itself — clicking "Simulate Attack" internally called `logAttemptForIP("192.168.1.50")` in a loop. No real network traffic, no real requests, no real block. The detection algorithm was sound but there was nothing genuine to detect. Additionally, several bugs were found: blocked IPs never auto-unblocked (contradicting the report's own test claims), the color-coded event log did not work (the `color` parameter in `appendLog` was never applied), and `buildDashboard()` called `buildMiddleRow()` twice — the second call created a dead object and the VGrow was never applied to the actual panel in the scene.

**What carried over:** The detection algorithm itself — `HashMap<String, LinkedList<Long>>` sliding window with amortised O(1) per request — is correct and is reused unchanged in the new approach.

## Current Approach — Real Server + Real Attack

### Concept

A Java HTTP server serves an actual login page. `ffuf` (a real penetration testing tool) hammers the `/login` endpoint with credential combinations. The server's detection engine processes each incoming HTTP request, tracks failed attempts per IP using the sliding window algorithm, and returns HTTP 429 when an IP is blocked — meaning `ffuf` receives hard rejections, not simulated ones. The detection is not a simulation; it is the server's actual request-handling logic.

### Architecture

```
┌─────────────────┐         HTTP POST /login          ┌──────────────────────────┐
│   ffuf (attacker)│ ────────────────────────────────► │                          │
│  (separate IP)   │                                   │   Java HTTP Server       │
└─────────────────┘         HTTP 429 (blocked)         │                          │
                   ◄────────────────────────────────── │  ┌─────────────────────┐ │
                                                        │  │  Detection Engine   │ │
┌─────────────────┐         HTTP GET /login            │  │                     │ │
│  Browser (user)  │ ────────────────────────────────► │  │  HashMap<String,    │ │
│  Login page      │                                   │  │  LinkedList<Long>>  │ │
└─────────────────┘         HTTP 200 (served)          │  │                     │ │
                   ◄────────────────────────────────── │  │  HashSet<String>    │ │
                                                        │  └─────────────────────┘ │
┌─────────────────┐         HTTP GET /status           │                          │
│  Browser         │ ────────────────────────────────► │  /login  — login page    │
│  Monitor page    │                                   │  /status — live JSON     │
└─────────────────┘         JSON (live IP states)      └──────────────────────────┘
                   ◄────────────────────────────────────
```

### Components

#### 1. Java HTTP Server (`Server.java`)
Built on `com.sun.net.httpserver.HttpServer` — part of the JDK since Java 6, zero external dependencies.

Three endpoints:

| Endpoint | Method | Purpose |
|---|---|---|
| `/login` | GET | Serve the login HTML page |
| `/login` | POST | Receive login attempt, run detection, return 200 or 429 |
| `/status` | GET | Return current IP states as JSON for the monitor page |

On every POST to `/login`:
1. Extract the real client IP from the request (`remoteAddress`)
2. Run `logAttemptForIP(ip)` — the sliding window algorithm
3. If IP is blocked → return `HTTP 429 Too Many Requests` immediately, no auth check
4. If credentials are wrong → return `HTTP 401 Unauthorized` and count the failure
5. If credentials are correct → return `HTTP 200 OK`

#### 2. Login Page (`login.html`)
A single static HTML file served by the Java server. Styled with Tailwind CSS via CDN — no build step, no npm, no bundler. A standard HTML form that POST-sends `username` and `password` to `/login`.

#### 3. Detection Engine (carried over from previous approach)
Same `HashMap<String, LinkedList<Long>>` sliding window. Extracted into its own class `DetectionEngine.java` with no JavaFX dependency so it can be unit-tested independently.

```java
// Core algorithm — unchanged from previous approach
loginAttempts.putIfAbsent(ip, new LinkedList<>());
LinkedList<Long> ts = loginAttempts.get(ip);
ts.addLast(System.currentTimeMillis());                    // Step 2: O(1) tail append
while (!ts.isEmpty() && now - ts.peekFirst() > TIME_WINDOW)
    ts.removeFirst();                                      // Step 3: amortised O(1) expiry
if (ts.size() >= THRESHOLD) blockedIPs.add(ip);           // Step 4: O(1) HashSet insert
```

Blocked IPs auto-unblock when the window expires: on every request the window is pruned first, then the threshold is checked fresh — so if no attempts arrive for `TIME_WINDOW` seconds, `ts.size()` drops to 0 and the IP is removed from `blockedIPs`. This fixes the bug in the previous implementation where blocked IPs were permanently stuck.

#### 4. Monitor Page (`monitor.html`)
A second static HTML page (also Tailwind-styled) that polls `/status` every second and renders a live table of IP states — CLEAN / WATCHING / BLOCKED — with hit counts. This replaces the JavaFX dashboard.

#### 5. The Attack — `ffuf`
`ffuf` is run from a **separate IP** (Docker container or VM — see Setup) against the `/login` endpoint with a wordlist of fake passwords. This generates real HTTP traffic. The demo sequence:

```
ffuf -w passwords.txt -X POST -d "username=admin&password=FUZZ" \
     -u http://<server-ip>:8080/login \
     -H "Content-Type: application/x-www-form-urlencoded"
```

After 5 failed attempts within 10 seconds, `ffuf` starts receiving HTTP 429. The monitor page shows the attacker IP flip to BLOCKED in real time.

### Data Structures

Same structures, now processing real input:

| Structure | Role | Complexity |
|---|---|---|
| `HashMap<String, LinkedList<Long>>` | Maps each client IP to its timestamp history of failed attempts | O(1) avg lookup & insert |
| `LinkedList<Long>` as FIFO Queue | Ordered timestamps per IP; head = oldest, tail = newest | O(1) head removal, O(1) tail insert |
| `HashSet<String>` | Blocked IP membership — checked on every incoming request before any auth logic | O(1) contains / add / remove |

---

### Complexity Analysis

| Operation | Average | Worst | Notes |
|---|---|---|---|
| HashMap get / putIfAbsent | O(1) | O(n) | Hash collision chain |
| LinkedList addLast | O(1) | O(1) | Direct tail pointer |
| LinkedList removeFirst | O(1) | O(1) | Direct head pointer |
| Window expiry loop | O(k) per call | O(k) per call | Each timestamp removed at most once → amortised O(1) |
| Full logAttemptForIP() | **O(1) amortised** | O(n+k) | n=collision chain, k=expired entries |
| HTTP 429 early-exit (blocked IP) | O(1) | O(1) | HashSet.contains() before any other work |

**Space:** O(I × T) where I = unique IPs tracked, T = max timestamps per IP (bounded by THRESHOLD).

### Setup and Demo Plan

#### Solving the Loopback IP Problem
ffuf and the server must have different IPs or the attacker is always `127.0.0.1`.

**Option A — Docker (recommended):**
```bash
# Start server container
docker run --rm -p 8080:8080 --name bf-server openjdk:21 java Server

# Run ffuf from a second container on the same Docker network
docker run --rm --network bridge ffuf/ffuf \
  -w passwords.txt -X POST \
  -d "username=admin&password=FUZZ" \
  -u http://bf-server:8080/login
```
Each container gets its own IP. The server sees the ffuf container's IP, not localhost.

**Option B — VM:**
Run the Java server on the host. Run ffuf inside a Linux VM (VirtualBox/VMware with bridged networking). The server sees the VM's IP.

#### Demo Sequence
1. Start `Server.java`
2. Open `http://localhost:8080/login` in a browser — confirm the page loads
3. Open `http://localhost:8080/status` — shows empty IP table
4. Open `monitor.html` in a second tab
5. Run ffuf from the separate IP
6. Watch the monitor page: the attacker IP escalates CLEAN → WATCHING → BLOCKED
7. ffuf output shows HTTP 200/401 responses, then HTTP 429 after the threshold
8. Manually submit a login from the browser — succeeds normally (different IP, unblocked)
9. Wait 10 seconds with no ffuf traffic — attacker IP reverts to CLEAN automatically
